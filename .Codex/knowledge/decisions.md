# Decisions

## 2026-07-16: 수강생 시트 결제수단은 운영 의미대로 분리해 반영한다
- 결정: 시트의 `랠리즈`, `카드결제/카드`, `현금영수증/현금`은 모두 `PAID`로 반영하고, `미납`과 기존 표기 `미결제`만 `PENDING`으로 반영한다.
- 이유: 결제수단 칸은 실제 수납 장부와 같아서, 납부 경로가 다를 뿐 랠리즈·카드·현금영수증은 모두 수납 완료다. 반대로 미납/미결제만 학부모 청구와 미납 알림 대상이어야 한다.
- 결정: 시트의 `이월`은 새 미납 청구를 만들지 않고, 같은 월에 기존 미납/연체 청구가 있으면 `Payment.status = CANCELED`, `method = CARRY_OVER`로 닫아 `취소/이월`로 표시한다.
- 이유: 이월은 부상·질병 등으로 지난달 수강료를 보전하는 기록이지 미납이 아니다. 기존 청구를 그대로 두면 관리자와 학부모 화면에서 실제로 내야 할 돈처럼 보이므로, 닫힌 청구로 처리해야 한다.
- 결정: `휴원`은 해당 월의 `PAUSED`, `퇴원`은 해당 월의 `WITHDRAWN` 이력으로 보존하고, 이후 월 데이터가 없는 것은 정상으로 본다.
- 이유: 학생이 그만둔 뒤 시트에 이후 월 행이 없는 것은 누락이 아니라 운영 상태의 결과다. 현재 인원 계산은 최신 월 기준으로 하고, 과거 월 상태는 학생 상세 히스토리에 남긴다.

## 2026-07-16: 관리자 신청/체험 긴 목록은 서버 페이지 단위로 나눠 로딩한다
- 결정: `/admin/apply` 첫 진입에서는 체험 문의 최근 50건과 수강신청 통계만 먼저 가져오고, 수강신청 목록은 해당 탭 진입 시 50건 단위로 요청한다.
- 이유: 선생님이 처음 보는 화면에는 오늘 처리할 요약과 최근 항목만 필요하다. 전체 신청/체험 데이터를 한 번에 가져오면 데이터가 누적될수록 첫 진입이 다시 느려진다.
- 결정: `getTrialLeads`와 `getEnrollApplications`는 기존 상태 필터 호출을 유지하면서 `limit/offset` 옵션도 지원한다.
- 이유: 기존 호출부를 깨지 않으면서 관리자 화면만 더 가볍게 만들 수 있다.
- 결정: 통계는 전체 기준으로 유지하고, 목록은 부분 로딩 상태를 인정한다. 필터 결과가 현재 묶음에 없어도 다음 묶음이 남아 있으면 “다음 50건에서 더 찾아보기”를 제공한다.
- 이유: 통계 숫자는 운영 판단에 필요하지만, 목록 전체를 첫 화면에 싣는 것은 속도에 불리하다.

## 2026-07-16: 신청/체험 연락 기록은 별도 이력 테이블로 분리한다
- 결정: 체험 문의와 수강신청에 연락 상태 컬럼을 계속 붙이지 않고, `ApplicationContactLog`에 연락 완료, 부재, 재연락 예약, 상담 메모를 누적한다.
- 이유: 한 신청 건에 여러 번 연락할 수 있으므로 원본 신청서 테이블은 서류철로 유지하고, 연락 이력은 메모지처럼 따로 쌓는 구조가 운영 히스토리를 잃지 않는다.
- 결정: 관리자 목록 조회에는 전체 로그를 싣지 않고 최신 연락 1건과 열린 재연락 1건만 LATERAL join으로 붙이며, 읽기 경로에서 테이블 생성/컬럼 보장 DDL은 실행하지 않는다.
- 이유: 관리자 속도 개선을 유지하려면 목록 payload와 DB 작업 수를 작게 유지해야 한다. 테이블 보장은 쓰기 액션과 운영 SQL에 맡기고, 아직 테이블이 없으면 기존 목록으로 fallback한다.

## 2026-07-16: 교사 가입은 관리자 개인 초대 링크로만 시작한다
- 결정: 공용 설치 화면이나 전화번호 조회로 교사 셀프가입을 열지 않고, 원장이 생성한 7일 유효 개인 초대와 전화 OTP를 유지한다.
- 이유: 설치 주소는 누구나 알 수 있으므로 여기서 교사 권한 계정을 만들게 하면 권한 오남용과 계정 열거 위험이 생긴다.
- 결정: SMS 발송 결과와 초대 생성 결과를 분리하고, 문자 실패 시에도 관리자가 개인 가입 URL을 복사·공유할 수 있게 한다.
- 이유: 문자 서비스 장애가 가입 진입점 자체를 없애지 않도록 하기 위해서다.

## 2026-07-15: 교사용 앱 설치 링크와 개인 초대 링크를 분리한다
- 결정: 반복 공유 가능한 설치 링크는 `/staff/install` 하나로 고정하고 초대 토큰을 manifest, 앱 시작 주소, 브라우저 저장소에 넣지 않는다.
- 이유: 초대 링크는 한 사람의 가입 권한을 가진 개인 열쇠이고 설치 링크는 여러 선생님에게 공유하는 안내 주소이므로 섞으면 토큰 노출과 재사용 문제가 생긴다.
- 결정: 설치 여부를 localStorage에 영구 기록하지 않고 현재 standalone 상태와 현재 세션의 설치 완료 이벤트만 사용한다.
- 이유: 앱 삭제 뒤 저장값만 남아 재설치를 막는 오판을 방지하기 위해서다.

## 2026-07-15: 교사용 앱은 `/staff` 범위의 독립 PWA로 제공한다
- 결정: 관리자 화면과 분리된 `/staff`를 공식 진입점으로 두고, 로그인도 `/staff/login` 주소 안에서 처리한다.
- 이유: 홈 화면에 설치한 앱에서 세션이 만료되어도 일반 브라우저 화면으로 튀지 않고 교사용 흐름을 계속 유지하기 위해서다.
- 결정: 교사용 수업·학생·청구 데이터는 오프라인 캐시에 저장하지 않는다.
- 이유: 휴대폰 분실이나 공용 기기 사용 때 학생과 학부모 개인정보가 기기에 남는 위험을 줄이기 위해서다.
- 결정: 카메라·마이크 권한은 교사용 경로에만 허용한다.
- 이유: 사진 등록과 음성 메모에는 필요하지만 관리자·공개 화면까지 권한 범위를 넓힐 이유는 없기 때문이다.

## 2026-07-15: 현재 수강 상태는 최신 월 시트 기준으로 맞춘다
- 결정: 2026년 전체 수강생 이력은 `StudentRegistrationLedger`에 계속 보존하되, 시간표와 현재 반 인원에 쓰는 `Enrollment.status = ACTIVE`는 최신 완료 배치의 최신 월만 기준으로 정합성을 맞춘다.
- 이유: 1월부터 7월까지 같은 학생이 반복 등장하는 것은 수강 이력이지 현재 인원이 아니다. 현재 반 인원은 “오늘 출석부”처럼 최신 월만 봐야 하며, 과거 월 ACTIVE를 그대로 두면 정원이 마감된 것처럼 보인다.

## 2026-07-15: 수납 원장 반영은 관리자 확인 후 적용한다
- 결정: 최신 시트 원장을 기준으로 수납 기록을 바로 자동 변경하지 않고, 관리자 수납/결제 화면에서 생성·수정·확인 필요 건수를 먼저 보여준 뒤 `변경 적용`을 눌렀을 때만 반영한다.
- 이유: 수강료 청구와 미납 알림은 돈과 직결되므로 휴원, 퇴원, 추가수강, 이월처럼 자동 판단이 애매한 행을 섞어 처리하면 잘못된 청구가 나갈 수 있다. 그래서 확실한 납부/미납 건만 자동 적용 대상으로 두고, 애매한 건은 관리자 확인 대상으로 남긴다.

## 2026-07-13: 월별 수강생 중복은 연락처 기반으로 우선 매칭한다
- 결정: Google Sheets 월별 데이터에 같은 학생이 반복 등장할 때는 아이 이름만으로 확정 연결하지 않고, 아이 이름과 학부모 연락처, 학생 연락처, 생년월일을 우선순위로 기존 `Student`와 매칭한다. 이름만 유일하게 맞는 경우는 약한 후보로 표시하되 자동 적용하지 않는다.
- 이유: 계속 다니는 아이들은 여러 월 시트에 반복되지만, 이름만으로 합치면 동명이인이나 형제 계정이 섞일 수 있다. 연락처와 생년월일은 학생을 구분하는 신분증 같은 역할을 하므로 자동 이관은 강한 근거가 있을 때만 진행하고, 애매한 행은 관리자 확인 대상으로 남기는 편이 운영 데이터가 안전하다.

## 2026-07-12: Move schedule operation from sheet cache toward DB-first slots
- Decision: Add `ScheduleSlot`, `ScheduleImportBatch`, and `ScheduleImportIssue` as the DB-first source for academy schedule operations. Keep `SheetSlotCache`, `ClassSlotOverride`, and `CustomClassSlot` as legacy inputs during migration, then import them into `ScheduleSlot`.
- Reason: The academy already runs enrollments, attendance, payments, trials, waitlists, and makeup sessions through DB tables, but the schedule source still depends on a sheet cache. A dedicated slot table makes the schedule the same kind of durable operational record as the rest of the academy system.
- Security note: New public-schema Supabase tables should not be assumed to be exposed automatically. The SQL setup enables RLS and revokes anon/authenticated Data API access until an explicit public policy is needed.

## 2026-07-12: 다크모드 브랜드 강조색은 주황이 아니라 라임으로 치환한다
- 결정: 라이트모드의 브랜드 강조색은 기존 주황을 유지하되, 다크모드에서는 `--brand-accent`를 `brand-neon-lime`으로 정의한다. 전역 안전망은 `brand-orange`/`orange` 배경·텍스트·테두리·hover·focus 누락분을 라임 계열로 바꾸고, 라임 배경 위 흰 글자는 남색 글자로 보정한다.
- 이유: 다크모드에서 주황을 단순히 어둡게 만들면 브랜드 체계가 흔들리고, 화면마다 갈색/주황 잔상이 남는다. 라이트=주황, 다크=라임이라는 규칙을 하나의 수도꼭지처럼 `--brand-accent`에 모으면 컴포넌트가 놓친 부분도 일관되게 따라간다.

## 2026-07-12: 브라우저/PWA 테마 색상도 앱 테마와 함께 움직인다
- 결정: 정적 `meta[name="theme-color"]`와 `manifest.json` 기본색은 다크모드 기준의 라임 강조색으로 두고, `ThemeColorUpdater`가 `next-themes`의 `resolvedTheme`에 맞춰 브라우저/PWA 헤더 색상을 런타임에 갱신한다.
- 이유: 홈페이지 기본 테마를 다크모드로 바꿔도 브라우저 헤더가 주황색으로 남으면 앱 전체가 밝은 모드처럼 보인다. 테마 색상은 앱의 외곽 프레임 색이므로, CSS 화면만이 아니라 브라우저가 읽는 메타 정보도 같이 맞춰야 한다.

## 2026-07-12: 다크모드 칩/뱃지 대비는 전역 안전망으로 보정한다
- 결정: 사이트와 관리자 전역에서 반복되는 `bg-surface-warm`, `bg-surface-section`, `bg-*-50/100`, `bg-*-50/50`, `text-*-700`, `border-*-200` 같은 밝은 surface와 pastel 상태 칩/뱃지는 `src/app/globals.css`의 `.dark ...:not([class*="dark:*"])` 안전망으로 보정한다. 이미 명시적인 `dark:bg-*`, `dark:text-*`, `dark:border-*`가 있는 컴포넌트는 건드리지 않는다.
- 이유: 같은 종류의 칩/뱃지가 사이트와 관리자 여러 화면에 퍼져 있어 페이지마다 수정하면 빠뜨리기 쉽다. 전역 안전망은 안전벨트처럼 빠진 곳만 잡아주고, 의도적으로 다크 스타일을 지정한 컴포넌트는 그대로 존중한다.

## 2026-07-12: 사이트 운영 점검 봇은 새벽 자동 실행과 수동 실행을 함께 둔다
- 결정: 사이트 운영 점검 봇은 관리자 대시보드 수동 버튼을 유지하면서, Vercel Cron으로 매일 UTC 17:00, 한국 시간 새벽 2시에 `GET /api/cron/site-ops-bot`가 자동 실행되도록 등록한다. cron endpoint는 기존 cron 패턴처럼 `CRON_SECRET` bearer 인증을 요구하고, 관리자 버튼과 같은 `runSiteOpsBot` 로직을 호출한다.
- 이유: 점검 자체는 매일 자동으로 돌아야 운영 누락을 빨리 발견할 수 있지만, 관리자 첫 진입 때 자동 실행하면 다시 느려질 수 있다. 새벽 2시는 선생님 사용량이 낮고, 기존 백업 cron이 KST 자정에 돈 뒤라 백업 상태까지 함께 확인하기 좋다.

## 2026-07-12: 사이트 운영 점검 봇은 관리자 첫 화면에서 자동 실행하지 않는다
- 결정: `/admin` 첫 진입의 사이드바 기본 탭은 사이트로 바꾸고, 관리자 화면에서는 대시보드 카드의 “점검 실행” 버튼을 눌렀을 때만 사이트 운영 점검 봇을 실행한다. 점검 봇은 DB 연결, 기본 설정, 공개 콘텐츠, 시간표, 신청 링크, 백업 저장소, 인스타 자동 게시 설정과 대기열을 점검한다. 자동 조치는 기본 설정 row 생성과 백업 bucket 생성처럼 되돌릴 위험이 낮은 항목으로 제한하고, 수동 확인이 필요한 항목은 관리자/부관리자 알림으로 남긴다.
- 이유: 운영 점검은 유용하지만 첫 화면 진입마다 백업 Storage, 인스타 큐, 여러 콘텐츠 카운트를 동시에 확인하면 관리자 속도가 다시 느려질 수 있다. 관리자 화면의 점검은 건강검진 버튼처럼 필요할 때 실행하고, 정기 점검은 새벽 cron에 맡기는 구조가 속도와 안정성의 균형이 좋다.

## 2026-07-12: 관리자 긴 목록은 화면과 DB 양쪽에서 점진화한다
- 결정: `/admin/students`는 초기 부분 데이터가 있으면 자동으로 전체 목록을 다시 가져오지 않고, 사용자가 “전체 목록 불러오기”를 눌렀을 때만 전체 원생을 조회한다. `/admin/trial`과 `/admin/apply`는 처음 50건만 렌더링하고 더 보기로 추가 표시한다. 수납/알림/피드백/체험/대기/보강/등록 통계처럼 관리자 화면에서 자주 쓰는 조회 패턴은 Prisma schema와 SQL 파일에 인덱스를 남긴다.
