## 최근 지식
- 원생 관리 최신 원생목록 점검은 시트 행 수와 실제 학생 수를 분리해서 보여준다. 한 학생이 여러 반을 들어 여러 행이 생겨도 운영 학생 수는 1명으로 계산해야 정원 마감 오해를 줄일 수 있다.
- 수강생 장부는 월별 반복 행을 같은 학생 ID에 귀속한다. 최신 월이 모두 휴원이면 현재 상태는 `PAUSED`, 과거 월 퇴원 후 이후 월 기록이 없으면 현재 상태는 `WITHDRAWN`으로 유지한다.
- 학생 상세 화면은 최신 완료 수강생 이관 배치의 `StudentRegistrationLedger`를 월별로 묶어 개인별 수강/결제/셔틀 히스토리를 보여준다.
- 학생 상세 화면의 운영 요약은 현재 상태, 최신 월 수강, 미납 금액, 청구서 상태와 납부 링크를 먼저 보여준다. 상담 중 필요한 정보는 긴 히스토리 표보다 위쪽 요약에서 바로 판단할 수 있어야 한다.
- 현재 수강 상태(`Enrollment.status = ACTIVE`)는 최신 완료 수강생 배치의 최신 월 기준으로 맞춘다. 2026년 전체 월별 반복 행은 학생별 히스토리로 보존하고, 시간표 인원에는 최신 월만 반영한다.
- 관리자 수납 월별 청구서는 `청구 대상 확인 → 청구서 생성` 흐름으로 분리한다. 생성 전 대상 학생, 금액, 기존 유지 건을 먼저 확인하고, 청구서 생성 시 학부모 SMS는 자동 발송하지 않는다.
- 수강생 시트 결제수단은 운영 의미로 해석한다. `랠리즈/카드결제/현금영수증`은 납부 완료, `미납/미결제`는 미납, `이월`은 부상·질병 등 결석 보전으로 기존 미납 청구가 있으면 `CANCELED/CARRY_OVER`로 닫는다.
- 현장 토스 단말기 결제는 온라인 PG 승인과 분리해 `PaymentTransaction.provider = TOSS_TERMINAL`로 기록한다. 단말기 승인번호는 `Payment.providerOrderId`와 감사 로그 metadata에 남겨 나중에 오프라인 거래내역과 대조할 수 있게 한다.
- 관리자 신청/체험 화면은 `slotKey`나 원문 에러 대신 요일·시간·반 이름, 처리 필요 배지, 쉬운 재시도 안내를 보여준다. 관리자가 볼 화면에는 개발 과정의 문구를 남기지 않는다.
- 관리자 신청/체험 목록은 학생명, 보호자명, 전화번호, 학교, 메모로 검색할 수 있어야 하며, 반복 처리 결과는 `alert`보다 화면 안 성공/오류 메시지로 남긴다.
- 관리자 신청/체험의 우선 처리 기준은 장기 대기(24시간 이상), 중복 연락처, 체험 후속 안내 미완료, 쌤 알림 미발송이다. 이 기준은 DB 상태 변경 없이 화면 계산으로 먼저 보여준다.
- 미납 알림은 청구서 생성과 분리된 별도 버튼으로 처리한다. 자동 발송보다 관리자 확인 후 발송하는 방식이 실사용 안정성이 높다.
- 학생 상세 화면에서는 수강 반별 `ACTIVE/PAUSED/WITHDRAWN` 상태를 직접 변경할 수 있다. 서버 액션에서 허용 상태값만 검증하고 관련 캐시를 무효화한다.
- 관리자 수납/결제 화면에는 최신 시트 저장 배치 기준 수납 대조 기능이 있다. 생성/수정/동일/확인 필요를 분리하고, 확인 필요 건은 자동 변경하지 않는다.
- 시간표는 `ScheduleSlot` DB 원본을 우선 사용한다. DB 데이터가 없을 때만 기존 시트 캐시로 fallback한다.
- 2026년 수강생 데이터는 스프레드시트 원본 보존 모델과 실제 `Student`/`Enrollment` 운영 모델을 함께 사용한다.

## 현재 요약
- DB는 Supabase PostgreSQL이고 Prisma를 사용한다.
- Supabase PgBouncer 호환 때문에 DB 구조 보강은 raw SQL 보장 패턴을 자주 사용한다.
- 공개 홈페이지와 관리자 페이지가 같은 Next.js 앱 안에 있다.
- 관리자 속도 개선 방향은 서버 초기 payload 축소, 지연 로딩, 서버 캐시, 불필요한 prefetch 제거를 우선한다.
- 스프레드시트는 2026년 7월 기준 이관 자료로 사용하고, 8월 이후 운영은 DB 중심으로 전환하는 방향이다.
