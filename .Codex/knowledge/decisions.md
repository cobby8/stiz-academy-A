# Decisions

## 2026-07-06: 프로젝트 정체성 확정
- 결정: 저장소는 STIZ 농구교실 다산점 홈페이지와 학원관리 플랫폼으로 본다.
- 이유: 실제 코드는 상품/장바구니/주문 쇼핑몰이 아니라 학원 운영 기능 중심이다.

## 2026-07-06: 메인 홈 갤러리 기준 데이터 통합
- 결정: 메인 홈 활동 사진 섹션은 `AcademySettings.galleryImagesJSON` 대신 공개 `GalleryPost` 데이터를 사용한다.
- 이유: `/gallery`와 메인 홈이 서로 다른 사진 저장소를 보면 관리자가 같은 사진을 두 번 관리해야 한다.

## 2026-07-06: 관리자 설정의 구 갤러리 입력 제거
- 결정: `/admin/settings`의 포토 갤러리 URL 입력은 제거하고 `/admin/gallery` 안내로 대체한다.
- 이유: 공개 홈과 `/gallery`가 이미 `GalleryPost` 기준으로 동작하므로, 더 이상 반영되지 않는 입력칸은 혼란을 만든다.

## 2026-07-06: 운영시간/정책/푸터/SNS 관리자 설정화
- 결정: 운영시간, 개인정보처리방침, 푸터 문구, SNS 링크는 `AcademySettings`에 저장하고 관리자에서 관리한다.
- 이유: 운영 문구는 자주 바뀔 수 있으므로 코드 배포 없이 관리자가 즉시 수정할 수 있어야 한다.

## 2026-07-06: 인스타그램 토큰은 환경변수로 분리
- 결정: Instagram Business Account ID와 자동 업로드 ON/OFF는 관리자 설정에 두되, API access token은 서버 환경변수에 둔다.
- 이유: access token은 민감 정보라 DB 텍스트 입력으로 저장하면 노출/백업 위험이 커진다.

## 2026-07-07: 인스타그램 가져오기는 공통 동기화 함수와 하루 1회 cron 사용
- 결정: 수동 버튼과 Vercel cron이 `syncInstagramGalleryPostsToDb`를 함께 사용하고, cron은 하루 1회 실행한다.
- 이유: 중복 저장, 캐시 갱신, 오류 처리를 한 곳에서 관리할 수 있고, 사용자가 하루 1회면 충분하다고 정했다.

## 2026-07-07: 선생님 인스타 자동화는 초안-승인 구조로 분리
- 결정: 선생님은 `/staff/quick-post`에서 AI 초안을 만들고, 실제 홈페이지/인스타 게시는 `/admin/gallery`에서 관리자 또는 부관리자가 승인한다.
- 이유: 여러 선생님이 활용하려면 업로드는 쉬워야 하지만, 외부 공개 게시물은 최종 검수 단계가 필요하다.

## 2026-07-08: 선생님 사진 업로드는 승인 없이 바로 게시
- 결정: 선생님은 본인이 만든 `/staff/quick-post` 초안을 직접 게시할 수 있고, 게시 시 홈페이지 갤러리에 먼저 반영한 뒤 인스타그램 게시를 시도한다.
- 이유: 운영 흐름상 매번 관리자 승인을 거치면 사진 업로드 속도가 떨어지므로, 선생님이 미리보기에서 문구를 확인하고 바로 게시하는 흐름을 기본으로 둔다. 관리자는 `/admin/gallery`에서 실패 또는 대기 초안을 보조 관리한다.

## 2026-07-08: 공지사항 홍보 발행은 갤러리와 분리
- 결정: 공지사항을 인스타 피드/스토리와 페이스북 광고 소재로 활용할 때는 `GalleryPost`를 만들지 않고 `SocialCampaignPost`에 별도 기록한다.
- 이유: 방학특강 같은 홍보 공지는 공개 갤러리 사진 기록과 성격이 다르다. 갤러리에 섞이면 학부모가 보는 활동 사진 흐름이 광고 게시물로 오염될 수 있으므로, 홍보 캠페인은 따로 관리한다.

## 2026-07-08: 이미지 저장 전 브라우저 압축 적용
- 결정: 주요 이미지 업로드는 서버 저장 전에 클라이언트 `canvas`로 JPG 리사이즈/품질 압축을 수행한다.
- 이유: 휴대폰 원본 사진을 그대로 저장하면 관리자와 공개 홈페이지가 원본 이미지를 많이 불러와 느려진다. 저장 전에 가볍게 만들면 이후 모든 화면의 이미지 다운로드 부담이 줄어든다.

## 2026-07-08: 인스타 게시는 홈페이지 갤러리 게시 후 후속 상태로 분리
- 결정: 선생님/관리자 초안 게시 시 `GalleryPost` 저장을 먼저 완료하고 `SocialPostDraft`를 `PUBLISHING`으로 둔 뒤 인스타그램 게시를 이어서 시도한다.
- 이유: Meta API 게시가 느리거나 일시 실패해도 홈페이지 갤러리 반영 성공 메시지는 즉시 보여줘야 실사용자가 안심하고 다음 작업을 할 수 있다. 남은 인스타 게시는 `PUBLISHED`/`FAILED` 또는 재시도 가능한 `PUBLISHING` 상태로 관리한다.

## 2026-07-08: 홈 히어로 공지는 가벼운 공개 목록으로 표시
- 결정: 홈 히어로에는 공개 공지의 제목, 고정 여부, 날짜만 최대 4개 표시하고 본문 HTML은 가져와도 렌더하지 않는다.
- 이유: 공지 노출은 첫 화면 정보성은 높이지만, 본문까지 렌더하면 모바일 첫 화면 DOM과 계산량이 늘어난다. 목록 링크만 제공하면 반응 속도를 유지하면서 공지 접근성을 높일 수 있다.

## 2026-07-09: 선생님 인스타 게시를 서버 큐와 cron으로 처리
- 결정: 홈페이지 갤러리 반영 후 `SocialPostDraft`를 `PUBLISHING` 상태로 두고, `/api/cron/social-posts`가 5분마다 1건씩 인스타 게시를 처리한다.
- 이유: Meta API 게시에는 미디어 처리 대기와 재시도가 있어 화면에서 기다리면 선생님/관리자 페이지가 느려진다. 서버 큐로 넘기면 사용자는 홈페이지 반영 성공을 즉시 확인하고, 브라우저를 닫아도 인스타 게시가 이어진다.

## 2026-07-09: 전역 폰트와 추적 스크립트는 첫 화면에서 가볍게 둔다
- 결정: 관리자에서 선택 가능한 Google 폰트들은 전역 등록을 유지하되 `preload: false`로 두고, Meta Pixel은 `NEXT_PUBLIC_META_PIXEL_ID`가 있을 때만 로드한다.
- 이유: 기존 빌드 산출물에서 홈 첫 화면에 폰트 preload 561개, 약 5MB가 잡혔다. 선택 후보 폰트를 모두 미리 받는 것은 체감 속도를 크게 해치므로, 실제 사용 시 필요한 폰트만 브라우저가 요청하게 한다.

## 2026-07-09: 전역 후보 폰트는 next/font/google에서 제거
- 결정: `src/app/layout.tsx`의 전역 `next/font/google` 등록을 제거하고, 관리자 폰트 선택값은 `src/lib/fonts.ts`의 CSS fallback 스택으로 처리한다.
- 이유: `preload: false`는 브라우저 선로딩만 막고, `next/font/google`은 여전히 빌드 시 Google Fonts와 `fonts.gstatic.com` 파일을 다운로드한다. 외부 네트워크 불안정이 빌드 전체를 막지 않게 하려면 후보 폰트를 빌드 단계에서 가져오지 않아야 한다.

## 2026-07-09: 런타임 폰트 stylesheet는 첫 paint 이후 로드
- 결정: Pretendard와 Material Symbols stylesheet를 전역 `<head>`에서 제거하고 `DeferredFontStyles` 클라이언트 컴포넌트가 첫 paint/idle 이후 삽입한다.
- 이유: 전역 stylesheet는 첫 화면 렌더를 막는 리소스가 될 수 있다. 앱의 기본 CSS와 레이아웃을 먼저 보여주고, 선택 폰트와 아이콘 폰트는 화면이 그려진 뒤 붙이면 홈페이지와 관리자 페이지 첫 반응이 가벼워진다.

## 2026-07-09: 학원 기본 설정은 서버 캐시와 태그 무효화를 사용한다
- 결정: `getAcademySettings()`는 `unstable_cache`로 5분 캐시하고, 관리자 설정 저장 시 `academy-settings` 태그를 즉시 무효화한다.
- 이유: 학원 설정은 거의 모든 공개 페이지와 일부 서버 작업에서 반복 사용되지만 자주 바뀌지 않는다. 요청마다 DB를 확인하는 대신 캐시를 두면 전체 페이지의 서버 부담이 줄고, 저장 시 태그 무효화로 최신성도 유지할 수 있다.

## 2026-07-09: 공개 페이지 플로팅 도구는 본체를 지연 로드한다
- 결정: 챗봇은 `ChatPanel`을 버튼 클릭 후 로드하고, 입학 가이드 투어는 `GuideTourLazyTrigger`가 버튼만 먼저 보여준 뒤 기존 투어 본체를 필요 시 로드한다.
- 이유: 챗봇 패널과 가이드 투어는 모든 방문자가 쓰는 기능이 아니지만 모든 공개 페이지에 붙어 있다. 첫 화면에는 버튼만 있으면 충분하므로, 큰 상호작용 코드는 클릭/투어 URL/첫 방문 예열 시점으로 미룬다.

## 2026-07-09: 홈 정적 섹션은 서버 컴포넌트로 렌더링한다
- 결정: `LandingPageClient`의 파일 단위 `use client`를 제거하고, 후기 캐러셀처럼 실제 브라우저 상호작용이 필요한 하위 컴포넌트만 Client Component로 유지한다.
- 이유: 홈 히어로, 신뢰 지표, 과정 안내, 갤러리 하이라이트, CTA는 대부분 정적 렌더링으로 충분하다. 전체를 클라이언트로 묶으면 첫 진입 JS가 커지므로, 서버에서 먼저 HTML을 만들고 작은 상호작용 섬만 hydration한다. 빌드 산출물 기준 홈 entry JS는 4개 합계 약 0.25MB로 확인했다.

## 2026-07-09: 공개 갤러리 라이트박스는 클릭 후 로드한다
- 결정: 갤러리 그리드는 서버 렌더링으로 만들고, 클릭 감지용 `GalleryLightboxController`만 초기 로드하며, 전체화면 `GalleryLightboxOverlay`는 동적 로드한다.
- 이유: 방문자는 갤러리 목록을 먼저 훑고 일부만 사진을 크게 본다. 라이트박스 키보드 이벤트, 스크롤 잠금, 큰 미디어 뷰어 코드를 첫 화면에 싣지 않아도 기본 탐색은 가능하다. 빌드 산출물 기준 `/gallery` entry JS는 4개 합계 약 0.25MB이며 라이트박스 본체는 초기 manifest에서 빠진 것을 확인했다.

## 2026-07-09: 관리자 공지 리치 에디터는 모달에서 로드한다
- 결정: `/admin/notices`는 `RichTextEditor`를 정적 import하지 않고, 새 공지/수정 모달이 렌더될 때 `next/dynamic`으로 로드한다.
- 이유: 선생님/관리자가 공지 목록을 확인하거나 소셜 발행을 누를 때는 리치 에디터가 필요 없다. 큰 편집기 번들을 작성/수정 순간으로 미루면 관리자 목록 초기 진입이 가벼워진다. 빌드 산출물 기준 `/admin/notices` entry JS는 5개 합계 약 0.27MB이며 `RichTextEditor`는 초기 manifest에서 빠진 것을 확인했다.

## 2026-07-09: 공개 헤더 인증 확인은 동적 컴포넌트로 분리한다
- 결정: `PublicHeader`는 Supabase 브라우저 클라이언트를 직접 import하지 않고, 계정 버튼과 로그아웃 UI를 `PublicAccountControls`로 분리해 동적 로드한다.
- 이유: 공개 헤더는 모든 공개 페이지의 첫 화면에 붙지만, 로그인 상태 확인은 본문 렌더링보다 덜 급하다. 인증 SDK를 초기 entry JS에서 빼면 첫 화면 다운로드 부담이 크게 줄어든다. 빌드 산출물 기준 홈 entry JS는 4개 265,411 bytes에서 3개 58,339 bytes로 줄었고, entry 파일에 Supabase 문자열이 없는 것을 확인했다.

## 2026-07-09: 공개 공통 아이콘은 Material Symbols를 우선한다
- 결정: 공개 헤더, 테마 토글, 챗봇 버튼의 단순 아이콘은 `lucide-react` import 대신 Material Symbols 텍스트 아이콘으로 렌더링한다.
- 이유: Material Symbols 폰트는 전역에서 이미 사용 중이므로, 첫 화면 공통 컴포넌트가 별도 아이콘 JS를 import할 필요가 없다. 빌드 산출물 기준 홈 entry JS는 58,339 bytes에서 56,093 bytes로 줄었고, entry 파일에 lucide 문자열이 없는 것을 확인했다.

## 2026-07-09: 관리자 shell은 서버 인증 정보를 재사용한다
- 결정: `/admin` 레이아웃의 `requireAdmin()` 결과에서 사용자 이름/이메일을 shell에 전달하고, `AdminShellClient`는 Supabase 브라우저 클라이언트로 사용자 정보를 다시 조회하지 않는다. 로그아웃은 서버 액션으로 처리하고, 알림/체험 카운트 API 호출은 첫 렌더 뒤로 지연한다.
- 이유: 관리자 레이아웃은 렌더링 전에 이미 인증을 통과한다. 같은 정보를 브라우저에서 다시 조회하면 공통 shell JS와 초기 네트워크 부담이 늘어난다. 빌드 산출물 기준 `/admin` entry JS는 2개 합계 약 0.05MB이며 Supabase 문자열이 없는 것을 확인했다.

## 2026-07-09: 업로드 이미지는 고유 URL 전제로 장기 캐시한다
- 결정: 로컬 fallback 업로드 `/uploads/:path*`에는 `Cache-Control: public, max-age=31536000, immutable`을 적용하고, Supabase Storage 업로드에는 `cacheControl: "31536000"`을 지정한다.
- 이유: 현재 업로드 경로는 타임스탬프와 랜덤값을 포함한 파일명으로 저장되어 같은 URL의 이미지 내용이 바뀌지 않는다. 한 번 받은 이미지를 브라우저와 CDN이 오래 재사용하면 홈페이지와 관리자 페이지의 재방문 이미지 로딩이 빨라진다.

## 2026-07-09: 공개 신청 페이지 HTML sanitize는 서버에서 수행한다
- 결정: `/apply`의 체험/수강신청 안내 HTML은 `page.tsx` 서버 컴포넌트에서 `sanitizeHtml()`로 정리한 뒤 `ApplyPageClient`에 전달한다.
- 이유: `sanitize-html`과 그 하위 파서들은 브라우저에서 실행하기엔 큰 도구다. 서버에서 한 번 정리해 안전한 HTML만 내려보내면 보안은 유지하면서 `/apply` client bundle에서 약 248KB짜리 HTML 파서 chunk를 제거할 수 있다.

## 2026-07-09: 최초 관리자 setup은 서버 API에서 처리한다
- 결정: `/setup` client는 Supabase 브라우저 SDK를 직접 import하지 않고, `/api/auth/check-setup` POST가 Auth 사용자 생성, `User` row upsert, 로그인 쿠키 설정을 처리한다.
- 이유: setup은 최초 1회 사용하는 화면이지만 기존 구조는 Supabase 브라우저 SDK 약 200KB를 첫 JS에 포함했다. 서버 API로 옮기면 같은 UX를 유지하면서 `/setup` client bundle을 크게 줄일 수 있다.

## 2026-07-07: `SocialPostDraft`는 raw SQL 보강 테이블로 둔다
- 결정: Prisma schema 마이그레이션 대신 `CREATE TABLE IF NOT EXISTS "SocialPostDraft"` 패턴을 사용한다.
- 이유: 기존 프로젝트가 Supabase PgBouncer 호환을 위해 raw SQL 보강 패턴을 많이 쓰고 있으며, 이번 기능은 독립 초안 테이블이라 점진 도입이 안전하다.

## 2026-07-07: 인스타 자동 게시 기본 모드는 사진 전용
- 결정: 사진 1장은 단일 게시, 사진 2~10장은 캐러셀 게시로 처리하고 영상 자동 게시는 이번 범위에서 제외한다.
- 이유: 사용자가 원하는 기본 흐름은 “사진만 넣으면 자동 생성/업로드”이며, 영상은 인스타 Graph API 조건과 처리 분기가 더 복잡하다.

## 2026-07-07: 로그인은 통합하고 회원가입에서만 계정 유형 선택
- 결정: 로그인 화면은 `STIZ 로그인` 하나로 통합하고, 회원가입 때만 학부모/선생님 유형을 선택한다.
- 이유: 로그인은 신분을 다시 묻는 곳이 아니라 인증하는 곳이다. 계정 유형과 권한은 DB `User.role`을 기준으로 판단하고, 로그인 후 역할별 기본 화면으로 이동시키는 편이 자연스럽다.

## 2026-07-08: 관리자 화면은 서버 레이아웃에서 권한 확인
- 결정: `/admin` 레이아웃은 서버 컴포넌트에서 `requireAdmin()`을 먼저 실행하고, 통과한 경우에만 클라이언트 관리자 shell을 렌더링한다.
- 이유: 미들웨어의 로그인 확인만으로는 일반 로그인 사용자가 관리자 화면을 볼 수 있다. 관리자 데이터 노출을 막으려면 페이지 렌더링 전 DB role 기준 권한 확인이 필요하다.
